**Key Implementation:**

- **Smart Cache Invalidation**: 3 triggers - (1) cache age >7 days, (2) earnings within 14 days, (3) **latest 3 earnings have null actuals** (detects when FMP fills in actuals)

**Usage:**

## 💡 Context/Memory

### Key Decisions Made
1. **Keep model_trainer.py** - Create new `model.py` instead of modifying existing file
2. **Backward compatibility** - All `M01_3BAR` references aliased to `M02`
3. **D3 column convention** - Triple barrier data uses `Date` (capital D), D2 uses `date`
4. **Survivor Model** - Uses MAE > -K×nATR filter, trains on y_max (MFE for survivors)
5. **Pre-generated reports** - Dashboard loads JSON files, not raw parquets

### Architecture Notes
- `M01` = Regression (predicts return %)
- `M02` = Classification (predicts ignition probability via triple barrier)
- Survivor model filters using `MAE > -K × nATR` (structural stop)
- Dashboard report pages load from pre-generated JSONs for speed

### Data Curator Architecture
- **Storage Pattern**: Single file for company profiles (cross-sectional lookup), per-ticker files for earnings/fundamentals/price (time-series data)
- **Earnings Calendar**: Per-ticker parquet files in `data/earnings/` with historical + future earnings
- **Smart Update Logic**: Compares `latest_earnings_date > fundamental_cache_mtime` to determine staleness
- **Retry Behavior**: Failed tickers automatically retry on next run (no cache file created on failure)

### Data Health Analyzer Architecture
- **Multi-Dimensional Analysis**: Price (200-bar filter), Fundamentals (quality + completeness), Profiles (key fields), Earnings (staleness detection)
- **Staleness Detection**: Compares earnings release dates against fundamental cache timestamps
- **Actionable Reporting**: Identifies stale caches and provides specific refresh recommendations
- **JSON Output**: Saves detailed results to `data/data_health_report.json` for programmatic access
