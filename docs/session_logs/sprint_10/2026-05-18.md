## 💡 Context/Memory

- **Review #1 (wiring bug)**: The plan originally wrapped Phase 1.5 with `if not phase_1_only:`, which would have silently skipped the gate in `--phase-1-only` mode — exactly the debugging mode where the coverage gate matters most. Inserted at L179 (before the early-return) with no guard.
- **Review #2 (dead config)**: Original plan added `"phase_1_5_quality_gate": PipelineFailureMode.WARN` to `PIPELINE_FAILURE_MODES`. That dict is only read inside `_execute_phase`/HALT path, which Phase 1.5 deliberately bypasses — an unread entry misleads future readers. Dropped entirely; non-blocking contract is enforced structurally and stated in docstring.
- **Review #3 (retry success path)**: The plan's retry sketch showed `# results[ticker] already False — no change needed`. This is only true for the failure path. The success path MUST flip `results[ticker]=True`, otherwise recovered tickers land in `price_data` but are still counted in `last_errors` by `update_cache:977`.
- **Review #4 (coverage definition)**: `_compute_price_coverage` uses exact-day presence; `audit_t1_data_quality.py` uses a 5-business-day staleness window. These differ by design — a same-run gate needs today's data, not "traded sometime this week". Step 1 centralises the threshold numbers only, not the coverage predicate. Documented explicitly in the docstring.
- **Logging root cause**: The orchestrator itself was never the noise source — it already had clean one-line-per-phase INFO output. The noise came entirely from (1) yfinance/urllib3 at INFO level (no suppression) and (2) 23 raw `print()` calls in `feature_pipeline.py` bypassing the level filter. Fix was surgical and touched zero pipeline logic.
- **DuckDB connection style**: File uses `conn = duckdb.connect(...) / try/finally conn.close()` — NOT the context-manager form. New helpers match this convention for consistency.
- **Current live coverage**: `_compute_price_coverage('2026-05-15')` = 98.56% (58 of 4176 active tickers missing exact-day row). Well above the 90% retry trigger — Phase 1.5 will be a sub-second no-op on normal days.
