## ✅ Accomplished

### Diagnosis
- Identified that Phase 6 takes ~16 min and Phase 7 takes ~100–800s (bimodal) from pipeline logs
- Root cause 1: `CREATE OR REPLACE VIEW` calls each end with a `COUNT(*)` probe that **forces full view execution during DDL** — the count for `v_d2_training` and `v_d3_deployment` traverses the entire chain (T2 session detection → T3 join → hydration → sl_exits)
- Root cause 2: `sl_exits` CTE in `v_d2_training` had **two correlated subqueries per SL-triggered trade** (`SELECT LIMIT 1` from `price_data`). With ~38K trades, this is O(n) serial seeks. Bimodal timing (100s vs 500–800s) was caused by DuckDB buffer pool state — fast when `price_data` pages were warm from Phase 5, slow when cold
- Confirmed row counts are stable (~38K) so the regression was not data volume — it was the correlated subquery becoming sensitive to buffer state

### Fix applied
- `v_d2_training`: replaced `sl_exits` correlated subqueries with a `price_with_next` CTE using `LEAD(date/close) OVER (PARTITION BY ticker ORDER BY date)` — one O(n) window pass instead of O(trades) random seeks
- Removed `COUNT(*)` probe after `v_d2_training` creation (was forcing full chain execution during Phase 6 DDL)
- Removed `COUNT(*), COUNT(DISTINCT ticker)` probe after `v_d3_deployment` creation (same issue — chains through `v_d2_features` → `v_d1_candidates` → full T2 scan)

### Validation
- Fix validated: `COUNT(*) = 38,243 rows in 202s`, `bad_exit_dates=None` — all SL exit dates are correctly after SL trigger dates
- 202s for cold COUNT is the `v_d2_hydrated` expansion cost (unavoidable) — the old correlated subquery added 400s on top; that is now eliminated

## 💡 Context/Memory

### Why COUNT probes are expensive in this view chain
DuckDB views are virtual (zero disk space, instant DDL). But any query against a view re-executes the full SQL from scratch. The COUNT probes at the end of each `_create_v_*` method were there to print status lines — harmless for cheap views (`v_sepa_candidates`), catastrophic for views that chain T2 session detection + T3 join + price_data expansion.

### The bimodal Phase 7 timing (100s vs 500–800s) explained
The correlated subquery in `sl_exits` did `SELECT LIMIT 1 FROM price_data WHERE ticker = X AND date > Y`. DuckDB executes this by probing the `price_data` B-tree for each trade. When `price_data` pages were already in the buffer pool (from Phase 5 T3 computation), seeks were fast (~100s total). When cold, each seek caused disk I/O → 5–8x slower. The `LEAD` window fix reads `price_data` once in a single sequential scan, making timing independent of buffer state.

### v_d2_hydrated is ML-only, not backtest
The backtester (`DuckDBCandidateFeed`, `UniverseScorer.score_from_duckdb()`) reads `t3_sepa_features` + `price_data` directly. `v_d2_hydrated` is only used by `v_d2_training`. This was confirmed by code search — no backtest file imports or queries `v_d2_hydrated`.
