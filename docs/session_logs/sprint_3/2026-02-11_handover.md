## 💡 Context/Memory

### Key Architectural Insights
1. **RS Rating is NOT Benchmark-Based**: Old design assumed `rs_vs_spy` (stock/SPY ratio). Actual implementation uses Minervini momentum formula: `0.4*ROC(63d) + 0.2*ROC(126d) + 0.2*ROC(189d) + 0.2*ROC(252d)`. This is pure performance-based ranking.

2. **Prices are Pre-Adjusted**: Current data does NOT have `adj_close` or `adj_factor` columns. All OHLC values are already adjusted for splits/dividends. Adding these columns to schema would create phantom columns.

3. **SEPA Filter is C1-C11 Rules**: Not just "price > SMA". Exact criteria from scanner.py:
   - C1-C9: Trend rules (Close > SMA_150/200, SMA_50 > SMA_150, rs_rank >= 0.70, etc.)
   - C10-C11: Breakout rules (Close > High_20D, Vol_Ratio > 1.0)

4. **Fundamentals Have Two Dates**: `report_date` (quarter end like March 31) vs `filing_date` (actual SEC filing like April 25). **MUST use filing_date for joins** to avoid lookahead bias. This is the #1 most critical fix.

5. **Market Cap Trap**: Using current `shares_outstanding` for historical market cap = inflated/deflated values after stock splits. A stock with 10M shares in 2020 that diluted to 50M shares in 2023 would show 5x inflated market cap for 2020 backtests. Phase 1 solution: Document as APPROXIMATE and warn against strict historical filters. Phase 2: Join with fundamentals for point-in-time shares.

6. **ROWS vs Calendar Days**: DuckDB window functions count trading rows, not calendar days. A "50-day MA" might span 64 calendar days if stock was halted. This is standard practice and acceptable for liquid US equities.

7. **MACRO Forces Date Filter**: Converting `v_sepa_candidates` from VIEW to MACRO prevents accidental full-table scans (millions of rows). Macro signature: `get_sepa_candidates(target_date DATE)` forces user to specify date.

8. **Earnings Can Be Rescheduled**: Companies sometimes reschedule earnings → duplicate (ticker, date) entries in source files. Must `drop_duplicates(subset=['ticker', 'date'], keep='first')` after sorting by `cache_timestamp DESC` to keep latest data.

### Design Decisions (Rationale)
- **Fundamentals PK = (ticker, report_date, filing_date)**: Allows multiple filings for same quarter (restatements) without overwriting historical data
- **RS Rank in Python (not SQL)**: Percentile ranking requires full cross-section per date. Easier in pandas via `df.groupby('date')['rs_rating'].rank(pct=True)` than complex DuckDB window functions
- **Backfill Script Separate**: Migration shouldn't block on cross-sectional calculations. Run as post-processing step
- **Phase 1 = Parallel Operation**: Files remain source of truth. DB is shadow for validation. Only switch to DB-primary after 1 week of 100% validation pass
