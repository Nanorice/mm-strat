## 💡 Context/Memory

### Architectural Decisions
1. **Why separate d2_rehydrated.parquet?** Preserves M01 backward compatibility - M01 continues using d2_features.parquet unchanged
2. **Why sequential trade rehydration?** Tried parallel but hit pickle errors with enriched_cache (DataFrames with thread locks). Sequential is fast enough (1,077 trades/sec).
3. **Why create FeatureEngine inside workers?** Avoid pickling thread locks - create fresh instances in each parallel worker process
4. **Why only trade_id?** User correctly identified other columns as either redundant (derivable) or premature (belong in backtest, not raw dataset)

### Triple Barrier Method (Deferred to Phase 2)
User initially requested Triple Barrier Method for M02 meta-labeling, but we agreed to **defer this to Phase 2** after backtesting exploration. This pragmatic approach allows us to:
- Explore trade characteristics first (MDD/MFE patterns)
- Test alternative exit strategies empirically
- Make informed decisions about barrier parameters based on data insights

**Triple Barrier concept** (for future reference):
- Upper barrier (profit target, e.g., +5%) → label = +1
- Lower barrier (stop loss, e.g., -5%) → label = -1
- Vertical barrier (time expiration, e.g., 5 days) → label = 0 or -1
- Labels based on which barrier touched FIRST (path-dependent)
- Barriers should be volatility-adjusted (dynamic based on ATR)

### 🏗️ Architecture Details

#### VectorBTBacktester (Single Trade Analysis)
**Purpose**: Analyzes individual trade trajectories using VectorBT's portfolio simulation

**Key Features**:
- Entry logic: Uses `open` price of first row (T+1 open, assuming T close = T+1 open)
- Exit logic: Uses `close` price of last row (actual exit date from d2_rehydrated)
- Column normalization: All columns converted to lowercase for consistency
- Metrics calculated:
  - Sharpe ratio, Sortino ratio, Calmar ratio
  - Max Drawdown % (MDD) - calculated from equity curve
  - Max Favorable Excursion % (MFE) - peak profit during trade
  - Total return %
  - Peak date and days to peak
  - Win rate (1.0 or 0.0 for single trade)
- Visualization: Equity curve and drawdown underwater plots

**Design Decision**: Error handling raises `ValueError` if exit_price is None (strict validation)

#### TradeAnalyzer (Batch Processing)
**Purpose**: Orchestrates VectorBTBacktester across all SEPA trades and generates comprehensive reports

**Key Features**:
- Loads d2_rehydrated.parquet with automatic column normalization
- Batch processing with progress bars (via tqdm)
- Portfolio storage for drill-down: `get_portfolio(trade_id)` retrieves full VectorBT Portfolio object
- Comprehensive reporting:
  - Overall statistics (avg return, Sharpe, MDD, MFE, days held, days to peak)
  - Winners vs Losers comparison (grouped by label)
  - Exit reason breakdown (grouped by exit_reason column)
  - **"Regret Analysis"**: Identifies trades with high MFE but poor final returns (exited too late)
- Automated visualizations (5 plots):
  1. Return distribution histogram (winners vs losers)
  2. Max drawdown distribution histogram
  3. Return vs MDD scatter plot
  4. Days to peak distribution histogram
  5. Holding period vs return scatter plot
- Results export to CSV

### 🎯 Key Implementation Decisions

1. **No Capital Allocation Logic**:
   - Current focus is exploratory analysis of individual trade outcomes, not portfolio simulation
   - User clarified: "we are just doing a prototype... the aim is to understand how price changes for SEPA trades"
   - Capital constraints deferred to future production backtester

2. **No Alternative Holding Period Testing**:
   - Initial proposal included testing different holding periods (5/10/20 days)
   - User corrected: Each trade already has actual entry/exit in d2_rehydrated
   - Focus is analyzing what ACTUALLY happened, not simulating alternatives

3. **Data Source Simplification**:
   - Initially proposed loading both d2 and d1 (trades metadata)
   - User clarified: All needed info already in d2_rehydrated (label, return_pct, days_held, exit_reason)
   - Simplified to single data source

4. **Entry/Exit Price Logic**:
   - Signals generated on T market close → enter on T+1 open
   - Assumption: T+1 open ≈ T close (realistic for liquid stocks)
   - Entry: `open` of first row, Exit: `close` of last row

5. **Slippage Modeling**: Ignored for now (can add later by adjusting entry/exit prices)

6. **Output Format**: DataFrame summary + method to retrieve full Portfolio objects (Option C - best of both worlds)
