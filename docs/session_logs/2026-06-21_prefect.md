## 💡 Context/Memory
- **DST is solved by the IANA tz name, not arithmetic.** Because both US and UK shift clocks ~together, US close ≈ 21:00 London year-round, so a fixed 22:00 London always lands ~1h after close.
- **`serve()` is the right shape for a single always-on box** (plays the worker role; no work-pool/worker ceremony needed). Two long-lived processes at boot: server (UI) + serve (scheduler).
- **The "two `run_daily_pipeline.py` processes" is a non-issue** — the orchestrator spawns subprocess phases (dashboard build, model card) as children; that's the 2nd process, not a 2nd run.
- **The "system Python" child was a red herring (corrected).** The orchestrator launches every subprocess phase with `sys.executable` (verified, lines 1312/1353/1432/1551), so phases run under `.venv`. Proof: system Python312 lacks `duckdb`/`xgboost`, yet the Model Card phase completed — it could only be `.venv`. What appeared as "system Python312" in the `ps` tree was a multiprocessing **spawn worker** surfacing under `sys._base_executable` (the venv's base interpreter). No fix needed.
- **`prefect deployment run` throws a cosmetic `UnicodeEncodeError`** rendering its banner to a cp1252 pipe — the run still triggers; set `PYTHONUTF8=1`.
- Prefect state lives in `~/.prefect` (SQLite, outside repo); launcher logs in `logs/prefect/` (gitignored).
- **Prefect CANNOT wake a sleeping machine** — its scheduler is just a user process. Waking is delegated to Task Scheduler `-WakeToRun` (wakes from sleep/hibernate S1-S4, NOT full shutdown S5 — that needs a BIOS RTC alarm + auto-login). Clean split: Task Scheduler wakes, Prefect sleeps.
- **Windows idle-sleep is driven by USER INPUT, not CPU load.** A long unattended run will NOT keep the box awake on its own — must hold `SetThreadExecutionState(ES_SYSTEM_REQUIRED)`. The flow holds it for the whole run (no fixed cap; tracks actual duration; released in `finally`). The 21:55 wake script's 10-min hold only *bridges* to the 22:00 flow start.
- **Real run is ~30 min** (fresh day: ingestion + T2/T3 + 764 MB dashboard build/upload). The 335s verification run was mostly idempotent skips — misleading as a runtime estimate.
- **Manual UI Quick-run of `daily` inherits `sleep_after=True`** and WILL sleep the box — use ad-hoc `python flows/daily_pipeline_flow.py` for a no-sleep manual run.
