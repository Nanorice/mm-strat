# Session Handover: 2026-03-31

## 🎯 Goal
Investigate data quality divergence between old `d1_trades.parquet` and current `v_d1_candidates` view, identify root causes, and fix bugs in SEPA trade entry/exit logic.

## ✅ Accomplished

### Bug Fixes
- **C11 volume spike fix** (`src/feature_pipeline.py`): `breakout_ok` was using `vol_avg_50` which included the current bar in the denominator — suppressing the volume ratio on breakout days. Fixed by adding `vol_avg_50_prev` (`ROWS BETWEEN 50 PRECEDING AND 1 PRECEDING`) and using it in `breakout_ok`. The old pipeline already did this correctly via `.shift(1).rolling(50).mean()`.
- **Session exit lookahead bias** (`src/managers/view_manager.py`): Exit was set to `last_trend_date` (last True day) — lookahead bias. Fixed to use `LEAD(date)` on `price_data` to get the first trading day *after* trend breaks. Also replaced correlated subqueries (`SELECT MIN(date) FROM price_data WHERE...` per trade) with `LEAD()` window function — eliminated OOM at 14GB.
- **Session boundaries used C1-C9** (`src/managers/view_manager.py`): C9 RS line flickers were fragmenting sessions (every C9 dip = exit + re-entry = 2 trades). Fixed to use C1-C8 only for session detection; entry still requires full `trend_ok` (C1-C9) + `breakout_ok`.

### Exit Criteria Refinement
- **Analysed which C1-C8 criterion actually drives exits**: C6 alone (close < SMA50) = 43.7%, C8 alone (>15% from 52W high) = 13.9%, C6+C8 together = 13.0%, C7 alone = 9.6%, C1/C2 true trend break = only 10.9%.
- **Conclusion**: C3/C4/C5/C7/C8 lag price by weeks/months and are not Minervini exit criteria. C8 and C7 were causing 27% premature exits.
- **Changed exit criteria to C1+C2+C6 only** (close > SMA150 AND close > SMA200 AND close > SMA50) in both `v_d1_candidates` and `v_screener_dashboard`. This is closer to Minervini's intent: exit on genuine SMA breakdown, not on 52W range metrics.

## 💡 Context/Memory

**Why old pipeline had more trades (14,448 vs ~13K new):**
Old pipeline (`FastTradeSimulator`) used `SEPA_Status = trend_ok (C1-C9)` as exit — every C9 RS line flicker caused exit + re-entry, fragmenting one long trade into multiple short ones. New pipeline uses C1+C2+C6, so C9 flickers and C7/C8 pullbacks are invisible to session boundaries.

**Old pipeline already had C11 correct**: `df['Volume'].shift(1).rolling(50).mean()` — the `.shift(1)` was the right fix. Our `vol_avg_50_prev` is the SQL equivalent.

**C3/C4/C5 never fire before C1/C2**: SMA crossovers (C3: SMA150>SMA200, C5: SMA50>SMA150) and SMA slope (C4: SMA200 rising) lag price by weeks. By the time they flip FALSE, C1 or C2 has already triggered. They are useful as *entry* filters (confirming established uptrend) but useless as exit signals.

**`screener_watchlist` is a materialized table**, not a view. It's rebuilt automatically by `ViewManager.create_all()` → `_refresh_screener_watchlist()`. No separate recompute step needed.

**LEAD() vs correlated subquery for exit date**: Original implementation used `SELECT MIN(date) FROM price_data WHERE ticker=X AND date > last_trend_date` per trade — O(n²), OOMed at 14GB for full history. Replaced with `LEAD(date) OVER (PARTITION BY ticker ORDER BY date)` precomputed once, then joined. Now runs in ~2 min.
