### Known issue ‚ÄĒ view-layer fan-out blocks end-to-end verification

When `feature_parity_check` runs against the live DB, the gate fails because
`v_d2_training` and `v_d3_deployment` each contain **multiple rows per
(ticker, date)** with inconsistent fundamentals (NaN vs value). `ROW_NUMBER() = 1`
arbitrarily picks one row and the two views' arbitrary choices don't agree.

**Implication.** XGBoost has been training on ambiguous data for months ‚ÄĒ the
training set silently fans out and the model averages over inconsistent
feature vectors. Until the view fan-out is fixed, every training run needs
`--skip-parity` and the metric should be flagged as
"trained-without-parity-verification."

**Investigation needed (separate session, ~half-day):** Find the JOIN against
`fundamentals` in `src/managers/view_manager.py` and add an as-of filter so
each (ticker, date) gets exactly one fundamental row. Probe queries OOM the
session ‚ÄĒ see saved memory `feedback_large_dataset_queries.md`.
