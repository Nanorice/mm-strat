- **Prefect server crash-loop ROOT-CAUSED + hardened** (`83a5245`). The night's recurring "server/serve keeps dying" was NOT (only) sleep/wake — captured the real error: `sqlite3.OperationalError: database is locked`. A single instance is stable. Fix: dropped `RestartCount`, added `MultipleInstances IgnoreNew`. **Re-registered (user, elevated) + restored to one server / one serve, healthy.**

## 💡 Context/Memory
- **Windows Sleep Reason "Application API"** (event log, Kernel-Power 42) = our `_suspend_pc()` hook; "Idle" would be the timeout (AC idle-sleep is **3h** here, so idle isn't the cause).
- **The "server/serve keeps dying" had TWO causes, untangled at the end:** (1) the box sleeping (suspends the processes) and (2) — the bigger one — **duplicate server instances on SQLite `database is locked`**. The `PrefectServer` task's `RestartCount=3` restarted the server while the old uvicorn child still held the lock → cascade. Hardened: no `RestartCount`, `IgnoreNew`. **A single server instance is stable.** Diagnose server death by running `python -m prefect server start` in the foreground to capture the real traceback (the launcher's Tee loses it).
- **`Stop-Process -Force` on the server can leave the SQLite WAL locked momentarily** — give it a few seconds before restarting, or the next start hits `database is locked`. (`~/.prefect/prefect.db` + `-wal`/`-shm` are normal WAL-mode files, not corruption.)
- **Clock jumps after sleep/wake**: PowerShell `Get-Date` and Python agreed, but jumped ~7h across the overnight sleep — the box was simply asleep that long, not a clock bug. A jumping wall-clock would still wreck cron, so worth a glance if scheduling misfires.
- **AtLogon tasks don't re-fire on wake-from-sleep** (only on logon) → the wake task must actively RESTART Prefect.
- **Scheduled run sleeps the box ~5s after completion** → server-side notifications can't deliver in time → notify must be **in-flow, before the sleep hook**.
- **Manual UI/CLI runs**: inherit `sleep_after=True` but `auto_scheduled=False` → never sleep. Safe to hand-trigger tests.
- **Dashboard**: reads open DuckDB fresh per call (no cached connection), so a swapped file takes effect immediately; only the import-time pull was the gap.
- "Two python processes" per role (server/serve/pipeline) = parent + spawned child — normal, not a duplicate.
- To test the scheduled-sleep path on demand: temporarily set the deployment schedule a few min ahead via `update_deployment_schedule` → creates an `auto_scheduled` run → revert after it starts Running (watcher keys on `state_name`, which stays "Scheduled" even when UI shows "Late").
