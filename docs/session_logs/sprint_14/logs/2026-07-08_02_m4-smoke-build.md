## 💡 Context/Memory
- **The session's biggest lesson (now a memory):** a training target is an INVARIANT — never fall back
  to a substitute column because the wanted one is missing. Decide scope by "would this make the model
  better?" FIRST, then enrich data to keep the target consistent. I twice reasoned from
  data-availability instead (fwd20-vs-mfe_pct; the population-expansion "not worth it"). Both corrected.
- **Read model IDENTITY from `model.json`, never prose** (carried from session 01) held up — I verified
  the champion is 4-class softprob against the artifact throughout.
- **The population-expansion finding is a real result, not just a decision:** eligible-non-breakout days
  are 100% post-breakout, return_20d 12.9% vs 8.2%, home-run 11.7% vs 4.0% — pure survivor/continuation.
- **Standards charter framing:** this is a RESEARCH-FIRST project with a production infra tail. Quality
  over cost for production code; result-HONESTY over cost for research. That split is why the ponytail
  laziness that's right for smoke scripts was WRONG when it bled into scoping decisions.
