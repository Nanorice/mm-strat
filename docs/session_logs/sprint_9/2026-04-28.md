## 💡 Context/Memory
- **Aha! Moment**: The error message `Invalid new DataFrame input for the: 49th feature...` was incredibly misleading. The Python XGBoost module doesn't bubble up the explicit C++ `kObject` vs `kInt32` type mismatch; it just generically complains about the "index type". Finding the `le.fit_transform()` mutation buried in the EDA phase of the notebook was the key to understanding why XGBoost expected integers despite `categorical_mapping.json` containing strings.
- **XGBoost Categorical Quirks**: If `enable_categorical=False` is passed to `xgb.DMatrix`, XGBoost bypasses pandas categorical index type validation entirely and treats everything as standard floats (which surprisingly "works" but is mathematically risky for tree splits). However, when `enable_categorical=True` is used, the dtype of the underlying Pandas Categorical `.categories` array (`object` vs `int64`) MUST exactly match the type used during `.fit()`.
