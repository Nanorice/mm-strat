## 💡 Context/Memory
- **serve vs server**: *server* = control plane (REST API + UI + SQLite `prefect.db`); passive, stores state. *serve* = worker/scheduler; registers the cron deployment, polls, and executes runs by shelling to `run_daily_pipeline.py`. Neither runs the pipeline alone.
- **Nightly chain**: `PrefectPipelineWake` (Task Scheduler, `-WakeToRun`, 21:45 Mon–Fri) wakes the box from **S3**, runs `wake_for_pipeline.ps1` which **kills and cleanly restarts both server and serve** (a slept server resumes broken — stale WAL/SHM → "database is locked"), holds awake to 22:00; the cron `0 22 * * 1-5` then fires the run; the flow's completion hook sleeps the box. The AtLogon boot tasks only cover the box while it's awake/logged in — the wake task is what re-establishes everything on a sleeping night.
- **Residual risk concentrates in 3 places**: (a) box is asleep-not-off at 21:45 (RTC does not wake from S4/S5), (b) the RTC wake actually fires (consumer-hardware ACPI is probabilistic), (c) no foreign process squats the `market_data.duckdb` write lock at 22:00 (single-writer — last night a stray `run_daily_pipeline.py` PID 8920 held it and crashed a sibling). Early warning = the wake script's Discord alerts.
