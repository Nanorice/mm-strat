## Daily pipeline integration

**Yes** — `sepa_watchlist` is updated daily by the orchestrator.

### Phase order (post-change)
```
1.   T1 Ingestion (price/fund/shares/macro)
2.   Screener Membership
3.   T2 Screener Features
4.   T2 Regime Scores (M03)
4b.  SEPA Watchlist Update           ← NEW
5.   T3 SEPA Features                ← now filters via sepa_watchlist
6.   View Refresh
7.   Training Cache Refresh
8.   Monitoring
```

### What Phase 4b does each day
`SepaWatchlistManager.update_daily(target_date)` runs three SQL operations:

1. **Close** any open session whose trend boundary breaks today (`close < sma_50 OR sma_150 OR sma_200`). Sets `exit_date`, `cooldown_end = exit_date + 14 days`, `status='COOLDOWN'`.
2. **Open** new sessions for tickers where `trend_ok AND breakout_ok` today AND no open session AND past prior session's cooldown. Inserts row with `status='ACTIVE'`, `session_id = max(prior) + 1`.
3. **Promote** any `COOLDOWN` row whose `cooldown_end < target_date` to `EXITED`. Refresh `trend_ok`/`breakout_ok` on remaining ACTIVE rows from today's t2 features.

Smoke-tested: 2026-05-06 update → +2 opened, 0 closed, 0 promoted, 416 active.

### Idempotency
`update_daily` is **not** idempotent across reruns of the same date — re-running it after a session closed on that date would see the now-COOLDOWN session as no-longer-open and try to open a duplicate. **Do not rerun Phase 4b on a date that has already been processed.** The orchestrator's existing per-phase idempotency guards prevent this in normal use.

If you ever need to re-run a date manually, drop today's events first:
```sql
-- Remove sessions opened today
DELETE FROM sepa_watchlist WHERE entry_date = '<DATE>';
-- Reopen sessions that were closed today
UPDATE sepa_watchlist
SET exit_date = NULL, cooldown_end = NULL, status = 'ACTIVE'
WHERE exit_date = '<DATE>';
```
Then re-run `update_daily('<DATE>')`.

## Architecture notes worth remembering

### Two watchlist tables — do not confuse
- **`sepa_watchlist`** (new, this session) — universe gate for the pipeline. Event log of SEPA sessions per ticker. Used by `compute_t3_features()` to filter the T3 row set. Updated by `Phase 4b`.
- **`screener_watchlist`** (existing) — materialised trade log for the dashboard. Output of `v_d1_candidates` aggregated to one row per trade. Updated by `Phase 6` via `ViewManager.create_all()`.

Both coexist and serve different purposes. They should never be conflated.

### Cooldown is a session-opening gate, not a row-inclusion gate
T3 carries the **full price history** of every ticker that has ever opened a SEPA session, regardless of whether the ticker is currently active, in cooldown, or long-since exited. The cooldown rule (14 calendar days after `exit_date`) only prevents `sepa_watchlist` from opening a *new* session too soon after the last one closed. It does not remove rows from T3.

### Why fundamentals were removed from T3
- T3 carried 4 fundamentals columns (`net_income`, `revenue`, `shares_outstanding`, `peg_adjusted`) duplicated across every (ticker, date) row — quarterly data scaled to daily granularity.
- `v_d2_features` was already computing them on demand via `LEFT JOIN fundamental_features ff ON ticker AND ASOF filing_date <= date`.
- Storage savings: ~4 columns × 9.4M rows = ~300MB freed.
- ASOF JOIN cost moved from per-INSERT to per-query — net win because views are queried less than they're inserted.

### O(n) per-quarter growth fix (from morning) is preserved
The redesigned `compute_t3_features()` keeps the morning's two critical bounds:
- `candidates` CTE: `AND p.date <= '{end_date}'` (else the t2 join scans full history per ticker)
- `with_velocity` CTE: `AND t2.date BETWEEN '{fetch_start}' AND '{end_date}'`

Without these, per-quarter wall time grows linearly with cumulative T3 size, not chunk size.
