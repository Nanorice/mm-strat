# Session Handover: 2026-04-09

## 🎯 Goal
Add operational observability to the pipeline: per-table write timestamps, structured error tracking, and clean up logging noise across the pipeline stack.

## ✅ Accomplished

- **`table_write_log` table**: One row per table, upserted after each successful write. Tracks `last_written_at`, `last_rows_written`, `last_phase`. Queryable from EDA notebooks to check data freshness without relying on `MAX(date)`.
- **`pipeline_error_log` table**: Append-only, one row per ticker failure event. Tracks `run_id`, `phase_name`, `error_type` (auto-classified), `affected_entity` (ticker), `error_detail`. Multiple runs per day handled via `run_id` FK — query last run with `WHERE run_id = MAX(run_id)`.
- **Error classification**: `PipelineRunManager.classify_error()` buckets raw exception strings into `RATE_LIMIT`, `TIMEOUT`, `NO_DATA`, `VALIDATION`, `AUTH`, `FETCH_FAILURE`.
- **`DataRepository.last_errors`**: Populated after each `update_cache()` call. `_update_cache_fmp` now returns `(results, buffer, failed_tickers)` — 3-tuple instead of 2.
- **`FundamentalEngine.last_errors`**: Same pattern — populated after `update_fundamentals()`.
- **Logging refactor (orchestrator)**: Consistent `[Pipeline]` / `[Phase N]` prefixes throughout. Duplicate failure logs removed (only `_execute_phase` logs them now). Phase 2/6/7 now have result summary lines. Phase 8 headers prefixed. `exc_info=True` on all `.error()` calls.
- **Logging refactor (data_engine)**: FMP progress every 25 tickers → DEBUG every 100. Staleness detail → DEBUG. yfinance summary → DEBUG. `update_cache` summary moved to DEBUG.
- **Logging refactor (fundamental_engine)**: 5 INFO lines per `update_fundamentals` call → zero at INFO (result reported by orchestrator). Earnings calendar progress → DEBUG.

## 💡 Context/Memory

- **Multiple runs per day**: `pipeline_error_log` links to `pipeline_runs.run_id` which is auto-incremented. Each pipeline invocation = new `run_id`. To query last run's errors: `WHERE run_id = (SELECT MAX(run_id) FROM pipeline_runs WHERE run_date = CURRENT_DATE)`.
- **DuckDB has no native UPSERT for `table_write_log`**: Used DELETE + INSERT pattern (not `INSERT OR REPLACE`) because DuckDB's `ON CONFLICT DO UPDATE` requires a PRIMARY KEY constraint — we have one but tested the delete+insert pattern as more explicit.
- **Why logs moved to DEBUG**: The FMP path was logging every 25 tickers at INFO — this was the main source of log noise on a 3981-ticker run. At DEBUG level these are still available if needed for debugging.
- **Staleness log showed 3949 stale**: This is expected on the first run of the day (before data is written). Second run should show ~16. Not a bug.
