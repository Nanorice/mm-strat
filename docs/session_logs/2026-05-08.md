## 🏗️ Architecture Decision: Option C — SEPA-Persistent Watchlist

### The gap vs current implementation
Current T3 (Option B) uses `screener_membership.is_active = TRUE` as the universe gate. This means:
- T3 ≈ T2 — almost all screener-active tickers (~2,500/day), no SEPA distillation
- ~70% of T3 rows are for tickers with `trend_ok=FALSE AND breakout_ok=FALSE` — noise for model training
- Per-ticker window features (`mom_21d`, `rsi_14`, ATR, returns, pct_chg deltas) are recomputed from `price_data` even though they already exist in T2 — pure duplication
- Fundamentals joined per-row into T3 — duplicates storage, could be joined at query time instead

### Intended design (Option C)
T3 universe = **all tickers that ever appeared in `sepa_watchlist`**, regardless of current status.
- A ticker enters `sepa_watchlist` when: `trend_ok=TRUE AND breakout_ok=TRUE` AND cooldown has expired
- A ticker's session exits when: C1+C2+C6 breaks (`close < sma_150 OR close < sma_200 OR close < sma_50`)
- After exit: 14 calendar day cooldown before a new session can open
- T3 carries **full history** for watchlist tickers — no add/remove logic on T3 itself
- The link is simple: `SELECT DISTINCT ticker FROM sepa_watchlist`

### `sepa_watchlist` event log schema
```sql
CREATE TABLE sepa_watchlist (
    ticker       VARCHAR,
    entry_date   DATE,         -- first day trend_ok=TRUE AND breakout_ok=TRUE in session
    exit_date    DATE,         -- NULL if still active; first day after C1+C2+C6 break
    cooldown_end DATE,         -- exit_date + 14 calendar days (NULL if active)
    session_id   INTEGER,      -- monotonic per ticker, 1-based
    trend_ok     BOOLEAN,      -- updated daily (current state)
    breakout_ok  BOOLEAN,      -- updated daily (current state)
    status       VARCHAR,      -- 'ACTIVE' | 'COOLDOWN' | 'EXITED' — updated daily
    PRIMARY KEY (ticker, entry_date)
)
```

### Session rules
- **Entry trigger**: `trend_ok=TRUE AND breakout_ok=TRUE`, no prior open session, and `today > prev.cooldown_end` (or no prior session)
- **Exit trigger**: C1+C2+C6 break — same as existing `v_d1_candidates` session logic (NOT full `trend_ok` — avoids C9 RS flicker causing premature exits)
- **Cooldown**: 14 calendar days (≈10 trading days) after `exit_date` before new session eligible
- **Re-entry**: new `session_id`, new row — prior session's `exit_date` is preserved
- **Daily update**: `trend_ok`, `breakout_ok`, `status` updated on every run for all open sessions

### T3 compute redesign (thin copy, not recompute)
```
T3 = SELECT * FROM t2 WHERE ticker IN (SELECT DISTINCT ticker FROM sepa_watchlist)
   + 9 TS alphas          (Python, per-ticker rolling — small population)
   + 2 vol_adj features   (Python, per-ticker rolling — small population)
   -- fundamentals:       JOIN at query/scoring time, NOT stored in T3
   -- per-ticker windows: carry from t2 directly (mom_21d, rsi_14, ATR, returns, pct_chg, etc.)
   -- XS alphas/ranks:    carry from t2 directly
```
The `candidates` + `per_ticker` + `with_velocity` CTEs (recomputing ~40 cols from `price_data`) are eliminated. `compute_t3_features()` becomes a thin SQL copy + small Python pass.

### Pipeline flow (canonical — do not deviate)
```
price_data (T1)
    ↓ ScreenerManager (price ≥ $15, vol ≥ 500K)
screener_membership
    ↓ compute_t2_screener_features()
t2_screener_features  (~2,500 tickers/day, all features)
    ↓ SepaWatchlistManager.update_daily()
sepa_watchlist  (event log: entry/exit/cooldown/status per session)
    ↓ compute_t3_features()  [thin copy: WHERE ticker IN sepa_watchlist universe]
t3_sepa_features  (all watchlist tickers, full history, + TS alphas + vol_adj)
    ↓ ViewManager.create_all()
views / d2_training_cache  (model training, scoring, dashboard)
```

## 💡 Context/Memory

- **O(n) growth was a query planning issue**: DuckDB does not push late `WHERE date BETWEEN` predicates through window function CTEs. Always bound scans at the earliest CTE, not the final SELECT.

- **Option B backfill script** (`scripts/backfill_t3_sepa_features.py`) can be reused as-is for Option C — it calls `compute_t3_features()` which will be redesigned. The checkpoint/resume/pre-chunk-DELETE logic is solid.

- **Cooldown is a signal filter, not a data gate**: T3 carries full history for all watchlist tickers regardless of cooldown state. Cooldown only prevents `sepa_watchlist` from opening a new session too soon after exit.

- **Fundamentals decision**: removing from T3 schema saves storage and eliminates the ASOF JOIN cost at insert time. The join moves to `v_d2_hydrated` / `v_d2_training` where it already partially exists. Verify M01 feature set still resolves correctly after this move.

- **`screener_watchlist` (existing)**: this is a materialisation of `v_d1_candidates` (trade log). It is NOT the same as `sepa_watchlist` (universe gate). Both will coexist — `screener_watchlist` is for the dashboard/backtest trade log; `sepa_watchlist` is for pipeline universe management.
